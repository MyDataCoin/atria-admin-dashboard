# Журнал аудита — что нужно от бэкенда

Админка должна показывать **реальный** журнал действий с сервера.
Сейчас это мок: `handleAddAuditLog` пишет запись в `useState` в браузере, на сервер
ничего не уходит, при перезагрузке страницы журнал обнуляется.

`GET /api/v1/audit` в API уже есть, но подключить его как есть **нельзя** — см. п.2.

---

## 1. Какие события должны попадать в журнал

Записи создаёт **бэкенд сам**, внутри соответствующих команд. Фронт ничего не пишет —
он только читает. Это принципиально: аудит, который пишет клиент, не является аудитом,
его можно подделать или просто потерять при закрытии вкладки.

| # | Событие | Когда | `eventType` |
|---|---|---|---|
| 1 | Создание объекта | `POST /properties` | `PropertyCreated` |
| 2 | Изменение объекта | `PATCH/PUT /properties/{id}` | `PropertyUpdated` |
| 3 | Публикация объекта | `POST /properties/{id}/publish` | `PropertyPublished` |
| 4 | «Скоро в продаже» | `POST /properties/{id}/announce` | `PropertyAnnounced` |
| 5 | Публикация новости/финотчёта | `POST /publications` | `PublicationPublished` |
| 6a | Поступил тикет | `POST /support/tickets` | `TicketOpened` |
| 6b | Тикет закрыт | `POST /support/tickets/{id}/close` | `TicketClosed` |

Заодно логично покрыть обратные операции, раз они уже есть в API
(`unannounce`, `complete`, `pause`/`resume`, `DELETE /publications/{id}`) — но это
второй приоритет, семёрка выше обязательна.

Важно: **тикет заводит инвестор, а не админ** (события 6a/6b). Значит `userId` в записи
— это не всегда сотрудник. Журнал должен уметь показывать и такие события, и не падать
на них.

## 2. Проблема с текущим AuditLogDto

Сейчас отдаётся:

```
entityType, entityId, eventType, dataJson, userId, occurredOnUtc
```

А таблица журнала в админке показывает: **кто** (имя), **что** (действие),
**детали** (человекочитаемый текст), **статус**, **время**.

Из текущего DTO нельзя получить ни имени пользователя (только `userId` — сырой uuid,
фронту пришлось бы джойнить по реестру и он всё равно не найдёт там инвесторов),
ни текста события (`dataJson` — это сырой снапшот payload'а, а не описание).

### Просьба: добавить в AuditLogDto три поля

```json
{
  "id": "uuid",
  "eventType": "PropertyPublished",
  "entityType": "Property",
  "entityId": "uuid",
  "occurredOnUtc": "2026-07-14T10:00:00Z",
  "userId": "uuid | null",

  "actorName": "Иван Петров",        // ← НОВОЕ: имя того, кто совершил действие
  "summary": "Объект «Вилла Сола Кабиати» опубликован", // ← НОВОЕ: готовый текст
  "severity": "success"              // ← НОВОЕ: success | warning | alert
}
```

- **`actorName`** — денормализованное имя (join по user/kyc), как уже сделали
  `propertyName` в `PublicationDto`. Для системных событий — `null` или `"Система"`.
  Для тикетов это будет имя инвестора.
- **`summary`** — готовая строка на русском. Собирать её на фронте из `dataJson`
  значит дублировать доменную логику в клиенте; текст должен приходить с сервера.
- **`severity`** — у журнала есть фильтр по критичности (Все / SUCCESS / WARNING /
  ALERT). Без этого поля фильтр нечем наполнить. Если решите не делать — скажите,
  уберу фильтр из UI, но лучше сделать.

`dataJson` оставьте как есть, он не мешает.

### Чего НЕ нужно

`ipAddress` — таблица его показывает, но это выдумка фронта (там сейчас
захардкожен один и тот же IP). Либо отдавайте реальный IP запроса, либо скажите, что
не будете — я уберу колонку. **Врать в журнале аудита хуже, чем не показывать поле.**

## 3. Эндпоинт

**`GET /api/v1/audit`** — уже есть, роль Admin/Compliance (сейчас без токена отдаёт 401,
это правильно).

Что нужно доделать:

- **Пагинация** — как в `/publications`: `page`, `pageSize`, ответ
  `{ items, page, pageSize, totalCount, totalPages }`. Журнал растёт быстрее всего
  остального, отдавать его целиком нельзя.
- **Сортировка** `occurredOnUtc DESC` (новые сверху).
- Фильтры `entityType`/`entityId` уже есть — оставьте.
- Желательно `eventType` и `severity` как фильтры, и `search` по `summary`/`actorName`
  (в UI есть строка поиска). Если поиск на бэке делать не хотите — скажите, оставлю
  клиентский по загруженной странице.

---

## 4. Что важно

- Записи **иммутабельны**: только append, никаких update/delete. В UI журнал так и
  подписан («неизменяемый журнал»), это должно быть правдой.
- Событие пишется **в той же транзакции**, что и само действие. Иначе объект
  опубликуется, а запись потеряется — и журнал начнёт врать.
- **CORS** — напоминаю, для прода нужны заголовки, в деве мы ходим через Vite-прокси.
